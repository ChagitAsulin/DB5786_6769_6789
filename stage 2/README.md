# דוח הפרויקט - שלב ב: שאילתות ואילוצים

## מטרת השלב
בשלב זה ביצענו תשאול של בסיס הנתונים של מערכת אבטחת מלון, הוספנו אילוצים חדשים, בדקנו טרנזקציות עם `ROLLBACK` ו-`COMMIT`, והוספנו אינדקסים לשיפור ביצועים.

## קבצים בתיקיית שלב ב
- `Queries.sql` - כולל 8 שאילתות SELECT, מתוכן 4 כתובות בשתי צורות, 3 שאילתות UPDATE ו-3 שאילתות DELETE.
- `Constraints.sql` - כולל 3 אילוצים חדשים בעזרת `ALTER TABLE` ובדיקות שגיאה לכל אילוץ.
- `RollbackCommit.sql` - כולל דוגמת ROLLBACK ודוגמת COMMIT.
- `Index.sql` - כולל 3 אינדקסים ובדיקות `EXPLAIN ANALYZE` לפני ואחרי.
- `backup2` - יש ליצור מגיבוי עדכני אחרי סיום השלב.

## שאילתות SELECT כפולות

### SELECT 1 - דוח משתמשים פעילים לפי כמות אירועים שדיווחו
**מוטיבציה:** במסך משתמשים או צוות אבטחה נרצה לראות אילו עובדים דיווחו על יותר אירועים בחלוקה לשנה וחודש.

**צורה A:** שימוש ב-`JOIN` בין users, roles, departments, incidents.

**צורה B:** שימוש בתתי-שאילתות עבור שם תפקיד ושם מחלקה.

**הבדל יעילות:** בדרך כלל צורת `JOIN` תהיה יעילה וברורה יותר כאשר צריך להביא נתונים מטבלאות קשורות רבות, כי מנוע PostgreSQL יכול לתכנן את הצירופים בצורה אופטימלית. תתי-שאילתות מתואמות עלולות להתבצע שוב ושוב לכל שורה, בעיקר בטבלאות גדולות.

**צילום נדרש:** צילום קוד הרצה + עד 5 שורות תוצאה.

### SELECT 2 - מצלמות עם ניסיונות גישה שנכשלו או נדחו
**מוטיבציה:** במסך מצלמות/יומני גישה נרצה לזהות מצלמות או אזורים בעייתיים.

**צורה A:** JOIN + GROUP BY.

**צורה B:** תתי שאילתות מתואמות לספירה ולתאריך אחרון.

**הבדל יעילות:** כאשר יש הרבה רשומות ב-`access_logs`, עדיף בדרך כלל לבצע JOIN ואגרגציה פעם אחת, במקום להריץ COUNT נפרד לכל מצלמה.

### SELECT 3 - אירועים עם עובדים משויכים ונכסי IT קשורים
**מוטיבציה:** במסך פרטי אירוע נרצה להבין האם האירוע טופל על ידי עובדים והאם יש נכסים קשורים לחקירה.

**צורה A:** JOIN על טבלאות קשר ואגרגציה.

**צורה B:** EXISTS ותתי שאילתות לספירת משויכים ונכסים.

**הבדל יעילות:** `EXISTS` יכול להיות יעיל כאשר רוצים רק לבדוק קיום, אך כאשר בכל מקרה צריך לספור ולהציג נתונים רבים, JOIN עם GROUP BY לרוב יהיה פשוט יותר להבנה ולעיתים יעיל יותר.

### SELECT 4 - דוח גיבויים לפי עובד, שנה וחודש
**מוטיבציה:** במסך ניהול גיבויים נרצה לראות אילו עובדים ביצעו גיבויים ובאיזו תדירות.

**צורה A:** JOIN ואז GROUP BY.

**צורה B:** CTE שמחשב קודם את סיכום הגיבויים ואז מצטרף למשתמשים.

**הבדל יעילות:** CTE משפר קריאות ומפריד בין שלב הסיכום לשלב הצגת פרטי המשתמש. בגרסאות חדשות של PostgreSQL המנוע יכול לבצע אופטימיזציה טובה גם ל-CTE, אך עדיין יש לבדוק עם `EXPLAIN ANALYZE`.

## 4 שאילתות SELECT נוספות

### SELECT 5 - נכסי IT בעייתיים במיקומים שבהם היו אירועים
**מוטיבציה:** מתאים למסך נכסים ומפת סיכונים. מאפשר לזהות נכסים במצב Offline/Maintenance/Retired שקשורים לאירועים.

### SELECT 6 - חודשים עם עומס גבוה של יומני גישה בעייתיים
**מוטיבציה:** מתאים לדשבורד יומני גישה ומראה מגמות לפי שנה וחודש. השאילתא משתמשת בפירוק תאריך ל-`YEAR` ו-`MONTH`.

### SELECT 7 - מצלמות שדורשות תחזוקה
**מוטיבציה:** מתאים למסך תחזוקת מצלמות. מזהה מצלמות עם פער גדול בין התקנה לתחזוקה אחרונה.

### SELECT 8 - עובדים עם הרבה כישלונות גישה
**מוטיבציה:** מתאים למסך Audit למשתמשים. מאפשר לזהות משתמשים עם דפוס בעייתי של גישה שנכשלה או נדחתה.

## ROLLBACK ו-COMMIT

בקובץ `RollbackCommit.sql` יש שתי הדגמות:

1. עדכון סטטוס מצלמה ואז `ROLLBACK` - לאחר הביטול הנתונים חוזרים לקדמותם.
2. עדכון סטטוס נכס ואז `COMMIT` - לאחר האישור הנתונים נשארים מעודכנים.

בכל הדגמה יש לצלם:
- מצב לפני.
- מצב אחרי העדכון.
- מצב אחרי `ROLLBACK` או אחרי `COMMIT`.

## אינדקסים

### idx_access_logs_status_time
נוסף על `access_logs(status, access_time)`.

**מוטיבציה:** טבלת `access_logs` גדולה יחסית, ולכן סינון לפי סטטוס ותאריך מופיע הרבה במסכי ניטור ודוחות.

### idx_incidents_reported_by_date
נוסף על `incidents(reported_by_user_id, reported_date)`.

**מוטיבציה:** מאפשר לשלוף אירועים לפי מדווח ולפי תקופה, מתאים למסך משתמשים/אירועים.

### idx_system_backups_type_date
נוסף על `system_backups(backup_type, backup_date)`.

**מוטיבציה:** מתאים למסך ניהול גיבויים שמסנן לפי סוג גיבוי ותקופה.

### הסבר תוצאות צפוי
לפני יצירת האינדקס ייתכן שנראה `Seq Scan`, כלומר PostgreSQL סורק את כל הטבלה. אחרי יצירת האינדקס ייתכן שנראה `Index Scan` או `Bitmap Index Scan`, בעיקר אם התנאי מסנן חלק קטן מהטבלה. אם הטבלה קטנה או אם התנאי מחזיר הרבה שורות, PostgreSQL עשוי עדיין לבחור `Seq Scan`, וזה תקין כי לפעמים סריקה מלאה זולה יותר משימוש באינדקס.

לאחר שכל הקבצים רצים בהצלחה, ליצור גיבוי בשם `backup2`:

```bash

```

אם שם הקונטיינר או בסיס הנתונים שונה, להחליף בהתאם.
